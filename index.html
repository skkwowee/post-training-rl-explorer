RLHF / PPO

Reinforcement Learning from Human Feedback Ouyang et al., 2022

How it works

RLHF is the original alignment recipe. It's a three-stage process:

  1. Supervised fine-tuning (SFT) on high-quality demonstrations
  2. Reward model training on human preference comparisons
  3. RL fine-tuning with PPO to maximize the reward model's score while staying close to the SFT policy

The key insight is using a learned reward model as a proxy for human preferences, then optimizing against it with PPO while penalizing divergence from the original model via a KL term.

Pseudocode

# Stage 1: Train reward model on human preferences for (x, y_w, y_l) in preference_data: loss = -log_sigmoid(R(x, y_w) - R(x, y_l)) # Stage 2: PPO training loop for epoch in range(num_epochs): # Sample from current policy prompts = sample_batch(dataset) responses = generate(policy, prompts) rewards = reward_model(prompts, responses) # Compute advantages using GAE advantages = compute_gae(rewards, values) # PPO update with clipping for mini_epoch in range(K): ratio = policy(responses) / old_policy(responses) clipped = clip(ratio, 1-ε, 1+ε) * advantages policy_loss = -min(ratio * advantages, clipped) kl_penalty = β * KL(policy ∥ ref_policy) loss = policy_loss + kl_penalty optimizer.step(loss)

Pros

  • Strong empirical results (GPT-4, Claude, etc.)
  • Online learning — generates its own training data
  • Can optimize complex, non-differentiable rewards
  • Well-studied algorithm with theoretical guarantees

Cons

  • Requires 4 models in memory (policy, ref, RM, critic)
  • Sensitive to hyperparameters
  • Reward model can be exploited (reward hacking)
  • Complex infrastructure and training instability

DPO

Direct Preference Optimization Rafailov et al., 2023

How it works

DPO's key insight: you don't need a separate reward model. The optimal policy under the RLHF objective has a closed-form solution that relates the reward to the log-ratio of the policy and reference model:

Substituting this into the Bradley-Terry preference model gives us a loss we can optimize directly on preference pairs, bypassing reward modeling entirely:

The parameter controls how much the policy can deviate from the reference. Higher = stronger preference signal but more divergence allowed.

Pseudocode

# DPO — no reward model, no RL! for (x, y_w, y_l) in preference_data: # Compute log-probabilities under current policy log_pw = log_prob(policy, y_w | x) log_pl = log_prob(policy, y_l | x) # Compute log-probabilities under frozen reference log_rw = log_prob(ref_policy, y_w | x) log_rl = log_prob(ref_policy, y_l | x) # The implicit reward margin margin = β * ((log_pw - log_rw) - (log_pl - log_rl)) # DPO loss: negative log-sigmoid of the margin loss = -log_sigmoid(margin) optimizer.step(loss)

Pros

  • No reward model needed — simpler pipeline
  • Stable supervised training (no RL instability)
  • Memory-efficient (only 2 models: policy + reference)
  • Strong theoretical foundation

Cons

  • Offline only — can't improve from its own generations
  • Still needs a reference model in memory
  • Requires paired preference data (more expensive to collect)
  • Can underfit when β is too low

KTO

Kahneman-Tversky Optimization Ethayarajh et al., 2024

How it works

KTO is motivated by prospect theory from behavioral economics: humans perceive losses more strongly than equivalent gains. Instead of preference pairs, KTO only needs binary feedback — thumbs up or thumbs down on individual outputs.

The key insight: in prospect theory, the value function is concave for gains and convex for losses, with losses weighted more heavily (loss aversion parameter ).

Pseudocode

# KTO — only needs binary labels, not pairs! # Compute reference point (expected KL over dataset) z_ref = mean([KL(policy(y|x) ∥ ref(y|x)) for x, y in data]) for (x, y, is_desirable) in binary_feedback_data: # Log-ratio relative to reference point log_ratio = log_prob(policy, y|x) - log_prob(ref, y|x) u = β * (log_ratio - z_ref) if is_desirable: loss = λ_D * (1 - sigmoid(u)) # push up else: loss = λ_U * (1 - sigmoid(-u)) # push down optimizer.step(loss)

Pros

  • Only needs binary feedback — much cheaper to collect
  • Grounded in human decision theory (prospect theory)
  • Works with unbalanced positive/negative data
  • Matches or beats DPO on many benchmarks

Cons

  • Still requires a reference model
  • Offline only — no online generation
  • Less established than DPO in production
  • Hyperparameter sensitivity (λ_D, λ_U, β)

GRPO

Group Relative Policy Optimization Shao et al., 2024

How it works

GRPO simplifies PPO by eliminating the critic (value) network entirely. Instead of learning a value function to estimate advantages, GRPO samples a group of outputs for each prompt and uses the group statistics as a baseline.

For each prompt , sample responses , score each with a reward model, then normalize within the group:

The policy is then updated using a PPO-style clipped objective with these group-relative advantages and a KL penalty to the reference policy:

Group Sampling & Advantages

Each resample generates a new group of 8 responses for a prompt. The purple bars show raw reward scores; green/red bars show normalized advantages. The dashed line is the group mean.

Pseudocode

# GRPO — no critic needed! Group statistics replace the value function for x in prompts: # Sample a group of G responses group = [generate(policy, x) for _ in range(G)] rewards = [reward_model(x, y) for y in group] # Compute group-relative advantages μ, σ = mean(rewards), std(rewards) advantages = [(r - μ) / σ for r in rewards] # PPO-style clipped update for y_i, A_i in zip(group, advantages): ratio = policy(y_i|x) / old_policy(y_i|x) clipped = clip(ratio, 1-ε, 1+ε) * A_i loss = -min(ratio * A_i, clipped) loss += β * KL(policy ∥ ref_policy) optimizer.step(loss)

Pros

  • No critic network — significant memory savings over PPO
  • Online learning from own generations
  • Works well with verifiable rewards (math, code)
  • Used in DeepSeek-R1, proving effectiveness at scale

Cons

  • Requires generating G outputs per prompt (compute cost)
  • Still needs a reward model for scoring
  • Group size G is a sensitive hyperparameter
  • Variance of advantage estimates depends on group quality

SimPO

Simple Preference Optimization Meng et al., 2024

How it works

SimPO removes the need for a reference model entirely. The key idea: use the average log-probability of a sequence under the policy itself as the implicit reward. This is both the reward signal and the optimization target.

The length normalization () is crucial — it prevents the model from favoring shorter responses simply because they have fewer terms in the sum.

SimPO also adds a target reward margin to ensure a minimum gap between preferred and rejected:

SimPO vs DPO Loss Comparison

The margin shifts SimPO's loss curve to the right, requiring a larger reward gap before the loss approaches zero. This encourages stronger separation between preferred and rejected outputs.

1.0

Pseudocode

# SimPO — no reference model, no reward model! for (x, y_w, y_l) in preference_data: # Length-normalized average log-probs as reward r_w = sum(log_prob(policy, y_w | x)) / len(y_w) r_l = sum(log_prob(policy, y_l | x)) / len(y_l) # SimPO loss with margin γ margin = β * r_w - β * r_l - γ loss = -log_sigmoid(margin) optimizer.step(loss)

Pros

  • No reference model — lowest memory footprint (1 model)
  • Simplest implementation of all methods
  • Length normalization prevents degenerate solutions
  • Margin γ provides a controllable quality gap
  • Outperforms DPO on several benchmarks

Cons

  • Offline only — requires pre-collected preference data
  • Newer method, less battle-tested in production
  • Average log-prob reward may not capture all quality aspects
  • Sensitive to the margin hyperparameter γ
